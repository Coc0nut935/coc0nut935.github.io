面试回答技巧：先说整体架构，再说细节；结合项目实际内容，不要之说理论
多级缓存（Caffeine + Redis）：Redis 分布式布隆过滤器 + 空值缓存防穿透；数据变更时Kafka异步广播本地缓存失效事件（独立消费组实现广播语义），缩短脏数据窗口期；秒杀扣库存通过Lua 脚本直接操作 Redis 保证原子性，实现读写链路隔离。
**面试官：先简单介绍一下，你这个项目中多级缓存的整体架构是怎样的？请求进来后，缓存的读取顺序是什么？**
回答：我们的多级缓存架构分为三层：L1 本地缓存（Caffeine）→ L2 分布式缓存（Redis）→ L3 数据库（MySQL）。
先查 L1 本地缓存，如果本地缓存未命中，就查 L2 Redis 缓存，
如果 Redis 也没命中就布隆过滤器拦截，如果布隆过滤器说不存在，直接返回，避免无效的数据库查询。
布隆过滤器通过后，再检查本地缓存和 Redis中是否有空值标记。如果有，说明之前已经查过数据库确认不存在，直接返回。
如果前面都没命中，加分布式锁，锁内再次检查本地缓存和 Redis（防止并发场景下其他线程已经回填了缓存），最后才查数据库。
数据库查到数据后，同时写入 Redis 和本地缓存；如果数据库也没有，写入空值缓存（本地 + Redis），TTL 设为 2 分钟，防止缓存穿透。
**面试官追问 1：为什么选择 Caffeine 而不是 Guava Cache 或 Ehcache？**
回答：Caffeine 的性能在 Java 本地缓存中是最优的，它的 W-TinyLFU 淘汰算法命中率接近理论最优。相比 Guava Cache 的LRU，Caffeine 在扫描密集型访问模式下命中率更高。相比 Ehcache，Caffeine 更轻量，不需要额外配置，且与 Spring Boot生态集成更好（Spring Cache 原生支持）。
Caffeine原理：[https://jishuzhan.net/article/2024659729461608450]()
**面试官追问 2：本地缓存和 Redis 缓存的数据一致性如何保证？**
回答：这个项目采用的是异步失效广播 + 短 TTL 兜底的方案：

1. 数据变更时，先删除 Redis 缓存，再通过 Kafka 向所有节点广播缓存失效消息。
2. 每个节点有独立的消费组，收到消息后删除本地缓存。
3. 本地缓存的 TTL 根据秒杀结束时间动态计算，但最短不会低于 1 秒，作为兜底保证最终一致性。

这种方案的脏数据窗口期取决于 Kafka 的消费延迟，通常在毫秒级别。相比 Redis Pub/Sub，Kafka的消息持久化能力更强，不会因为节点重启丢失消息。

**面试官追问 3：为什么不在数据变更时直接删除本地缓存，而要通过 Kafka 广播？**
回答：因为本地缓存是每个 JVM 实例独立的，数据变更可能发生在任意一个节点，但其他节点的本地缓存并不知道。如果不广播，其他节点会继续返回旧数据。通过 Kafka 广播，所有节点都能收到失效通知，保证集群级别的缓存一致性。
**面试官追问 4：布隆过滤器的误判率你设置的是多少？如何初始化的？**
回答：我们设置的误判率是 1%，预期插入量是 2 万条。布隆过滤器在应用启动时通过 BloomFilterDataInit初始化，从数据库加载所有有效的优惠券 ID 加入布隆过滤器。布隆过滤器的特点是宁可误判存在，不会误判不存在，所以它能有效拦截不存在的 key，防止缓存穿透。
**面试官追问 5：如果布隆过滤器误判了，会发生什么？**
回答：如果布隆过滤器误判一个不存在的 key 为存在，请求会继续往下走到空值缓存检查和数据库查询。最终数据库会返回空，然后写入空值缓存。下次同样的请求会直接被空值缓存拦截。所以布隆过滤器的误判只会导致少量请求穿透到数据库，不会造成严重问题。

第二轮：布隆过滤器

面试官：你提到了"Redis 分布式布隆过滤器防穿透"，我想深入问几个问题：

1. 布隆过滤器的原理是什么？它有什么特点？
2. 为什么选择布隆过滤器而不是其他方案（比如空值缓存、数据库索引）来解决缓存穿透？
3. 布隆过滤器存在误判，你是如何处理这个问题的？误判率是如何控制的？
4. 在你们的项目中，布隆过滤器的数据是如何初始化的？当数据库数据变更时，布隆过滤器如何更新？

**面试官：布隆过滤器的原理是什么？它有什么特点？**
回答：
原理：布隆过滤器是一个基于多个哈希函数 + 位数组的概率型数据结构。
1.初始化：创建一个长度为 m 的位数组，全部初始化为 0。
2.添加元素：当一个元素要加入集合时，用 k 个不同的哈希函数对该元素进行哈希，得到 k 个位置，将位数组中这 k个位置全部置为 1。
3.查询元素：判断一个元素是否在集合中时，同样用这 k 个哈希函数计算位置，如果所有位置都是1，则认为元素可能存在；如果有任意一个位置是 0，则元素一定不存在。
特点：
1.可能存在误判（False Positive）：因为哈希冲突，不同元素可能映射到相同的位，所以布隆过滤器说"存在"时，元素可能实际上不存在。但它绝不会漏判（No False Negative）——如果元素真的存在，布隆过滤器一定会说存在。
2.空间效率极高：不需要存储元素本身，只存储位数组。比如我们项目中，10 万个优惠券 ID，误判率 1%，只需要约 1.2MB内存。
3.查询时间 O(k)：与集合大小无关，只与哈希函数个数有关，非常快。
4.不支持删除：因为一个位可能被多个元素共享，置 0 可能影响其他元素。如果需要删除，得用 Counting BloomFilter（每个位用计数器代替）。
**面试官追问 1：你在项目中是怎么用布隆过滤器的？为什么选择它？**
回答：我们项目用布隆过滤器来防缓存穿透。场景是：秒杀优惠券查询时，如果传入一个不存在的voucherId，请求会穿透缓存直达数据库，恶意请求可能导致数据库被打崩。项目的实现是基于 Redisson 的 RBloomFilter，数据存储在 Redis 中，所以是分布式的——所有节点共享同一个布隆过滤器。
**面试官追问 2：为什么误判率设置为 1%？可以更低吗？**
回答：误判率越低，需要的位数组越大，哈希函数越多，内存占用和计算开销都会上升。1% 是一个经验值，在我们的场景下，即使误判了，请求也只是穿透到数据库，然后数据库返回空，写入空值缓存，下次就不会再穿透了。所以 1% 的误判率对系统影响很小，但能节省大量内存。如果设置为 0.1%，位数组大小大约翻倍，内存占用从 1.2MB 变成 2.4MB，收益不大。
**面试官追问 3：布隆过滤器的数据什么时候更新？新增优惠券怎么办？**
回答：我们在应用启动时一次性加载。如果运行期间新增了优惠券，有两种处理方式：
1.主动添加：在优惠券创建时，同步调用 bloomFilterHandler.add() 将新 ID 加入布隆过滤器。
2.重启更新：如果新增频率很低，可以依赖应用重启时重新初始化。
我们项目目前是启动时初始化。如果新增优惠券，布隆过滤器中没有这个ID，查询时会直接返回"不存在"。但实际上优惠券是存在的，这属于漏判——但布隆过滤器理论上不会漏判，所以这种情况说明我们的初始化逻辑需要优化，应该在优惠券创建时同步更新布隆过滤器。
**面试官追问 4：布隆过滤器存储在 Redis 中，如果 Redis 挂了怎么办？**
回答：如果 Redis 挂了，布隆过滤器不可用，此时有两种策略：
1.降级跳过布隆过滤器：直接走空值缓存 → 数据库的流程，相当于没有布隆过滤器，但系统仍然可用。
2.直接报错：如果对一致性要求很高，可以直接返回错误，避免请求穿透到数据库。
我们项目中，如果 Redis 不可用，整个缓存层都会失效，所以布隆过滤器不可用只是其中一个问题。实际上 Redis是高可用部署的（Sentinel 或 Cluster），单点故障的概率很低。
**面试官追问 5：布隆过滤器和空值缓存的区别是什么？为什么两个都要用？**
回答：
布隆过滤器：哈希计算，概率性、内存占用率极低（位数组）、可能误判存在、永久有效（除非主动删除）
空值缓存：实际查询结果，确定性、内存占用率较高（存储空值 key）、无误判、有 TTL，会过期
两者配合使用：
布隆过滤器：快速拦截肯定不存在的请求，内存开销极小。
空值缓存：缓存实际查询过但不存在的结果，TTL 兜底，防止布隆过滤器误判后的穿透。
布隆过滤器是第一道防线，空值缓存是第二道防线，两者互补。
**面试官追问 6：如果优惠券下架了，布隆过滤器怎么处理？**
回答：这是布隆过滤器的一个局限——不支持删除。如果优惠券下架了，布隆过滤器中仍然有这个ID，查询时会认为它"可能存在"，然后穿透到数据库。
在我的项目中使用不处理方案：下架的优惠券查询时会穿透到数据库，但数据库会返回空，然后写入空值缓存。下次请求就被空值缓存拦截了。这种穿透是可接受的。

  面试官：你提到了"空值缓存防穿透"，我想问：

1. 空值缓存的 TTL 你设置的是多少？为什么选择这个值？
2. 空值缓存和布隆过滤器在防穿透上是如何配合的？它们的职责边界是什么？
3. 如果一个恶意请求故意查询大量不存在的 key，空值缓存会不会导致 Redis 内存被大量占用？你如何应对这种情况？

**面试官：空值缓存的 TTL 你设置的是多少？为什么选择这个值？**

回答：我们设置的 TTL 是 2 分钟，定义在常量 CACHE_NULL_TTL = 2L，单位是分钟。
太短（比如几秒）：如果恶意请求持续攻击，几秒后缓存过期，请求又会穿透到数据库，防穿透效果差。

太长（比如几小时）：如果这个 key对应的数据在数据库中被创建了（比如新增了一张优惠券），但空值缓存还没过期，用户查询会返回"不存在"，造成数据不一致。

2 分钟：在我们的业务场景中，优惠券的创建频率很低（运营手动创建），2分钟的延迟对用户体验影响很小，但能有效拦截短时间内对同一不存在 key 的重复攻击。
另外，我们本地缓存和 Redis 都存了空值，TTL 都是 2 分钟。本地缓存的空值是通过设置 endTime 实现的，利用 Caffeine的过期机制自动淘汰。

**面试官：空值缓存和布隆过滤器在防穿透上是如何配合的？它们的职责边界是什么？**

回答与前面这两个区别相似

**面试官：如果一个恶意请求故意查询大量不存在的 key，空值缓存会不会导致 Redis 内存被大量占用？你如何应对这种情况？**

回答：会的，这是一个潜在风险，但我们的设计中有几层防护：

第一层：布隆过滤器前置拦截 -- 第二层：空值缓存的 key 有固定前缀 -- 第三层：TTL 自动过期 -- 第四层：限流兜底

其中我们的空值缓存 key 格式是 seckill:voucher:null:{voucherId}，value 只是一个很短的字符串"这是一个空值"，内存占用很小。一个 key 大约占用 100 字节左右，即使有 1 万个空值缓存，也只占用约 1MB。并且空值缓存 2 分钟后自动过期释放，不会永久占用内存。

**面试官追问 1：空值缓存在本地缓存和 Redis 都存了，会不会有数据不一致的问题？**

回答：不会。因为空值缓存的语义是"这个 key 不存在"，它本身就是一种兜底策略。即使本地缓存和 Redis 的过期时间有微小差异，最坏的情况就是多查一次数据库，不会返回错误数据。而且我们的空值缓存 TTL 都是 2 分钟，Caffeine 和 Redis 的过期机制都是可靠的，不会出现大的偏差。

**面试官追问 2：如果空值缓存过期了，但数据还是不存在，会不会又穿透到数据库？**

回答：会的。但这正是 TTL 存在的意义——允许一定的穿透，换取数据一致性。如果空值缓存永不过期，那数据被创建后，用户永远查不到。2 分钟的 TTL 意味着最多每 2 分钟会穿透一次到数据库，这个频率是可以接受的。

**面试官追问 3：你的本地缓存、Redis 缓存、空值缓存都是直接存一个 null 进去吗？如果不是为什么不直接存 null？**

回答：不是，三种缓存都没有直接存 null，存储方式各不相同：

**本地缓存**的类型是 Cache<String, SeckillVoucherFullModel>，Caffeine 本身不支持存 null 值，调用 cache.put(key, null)会抛异常。

所以我们的设计是：

未命中（本地缓存中没找到）：getIfPresent(key) 返回 null，表示缓存中没有这个 key

正常值：存 SeckillVoucherFullModel 对象

空值标记：存一个非 null 的 SeckillVoucherFullModel 对象，但只设置了 endTime 字段（设为当前时间 + 2分钟），其他字段都是 null

**Redis 缓存** —— 存的是 JSON 字符串，key 不存在表示未命中

Redis 使用 StringRedisTemplate，所有值都是 String 类型。

未命中（本地缓存中没找到）：opsForValue().get(key) 返回 null，表示 Redis 中没有这个     key

正常值：存 SeckillVoucherFullModel 的 JSON 序列化字符串，如{"voucherId":1001,"shopId":1,"stock":100,...}

空值标记：存一个独立的 key，格式是 seckill:voucher:null:{voucherId}，value 是字符串 "这是一个空值"，TTL 2 分钟

  第四轮：本地缓存失效

  面试官：你提到"数据变更时 Kafka 异步广播本地缓存失效事件"，这里有几个问题：

1. 为什么选择 Kafka 而不是 Redis 的 Pub/Sub 来做本地缓存失效广播？两者各有什么优缺点？
2. 你提到"独立消费组实现广播语义"，能解释一下这是什么意思吗？为什么不直接用 Kafka 的广播模式？
3. 如果 Kafka 消息丢失了，本地缓存的脏数据窗口期会变长，你提到"短 TTL 兜底最终一致性"，这个 TTL
   你设置的是多长？是如何权衡的？
4. 在高并发场景下，如果多个节点同时收到缓存失效消息，会不会出现"惊群效应"？你是如何处理的？

**面试官：为什么选择 Kafka 而不是 Redis 的 Pub/Sub 来做本地缓存失效广播？两者各有什么优缺点？**

回答：

1.消息持久化：Kafka 的消息会持久化到磁盘，即使消费者暂时离线，重新上线后仍然可以消费到之前的消息。我们的消费者配置了auto-offset-reset: latest，并且使用手动 ack，确保消息不会丢失。

2.消息的可靠性保障：

```
 producer:
   acks: all           # 等待所有副本确认
   retries: 5          # 失败重试 5 次
   properties:
     enable.idempotence: true  # 幂等生产者，防止重复消息
```

而且在 afterSendFailure 中实现了指数退避重试 + 死信队列兜底，即使发送失败也不会丢消息。

3.消费者组语义：Kafka 原生支持消费者组，可以灵活控制消息的分发模式。我们通过给每个实例分配不同的 groupId 来实现广播语义，这在 Redis Pub/Sub 中需要自己管理订阅关系，复杂度更高。

**面试官："独立消费组实现广播语义"是什么意思？**

回答：

核心机制：每个应用实例使用唯一的 groupId

```
@KafkaListener(
    topics = {"${prefix.distinction.name}-seckill_voucher_cache_invalidation"},
    groupId = "${prefix.distinction.name:hmdp}-seckill_voucher_cache_invalidation-${random.uuid}"
)
```

Kafka的语义是：同一个消费者组内的多个消费者，消息只会被其中一个消费；不同消费者组之间，每条消息都会被每个组消费一次。

所以当每个实例的 groupId 都不同，它们属于不同的消费者组，所以每个实例都会收到全量的缓存失效消息。这就是广播语义。

**面试官：Kafka 消息丢失了怎么办？TTL 设置多长？**

回答：

我们项目对 Kafka 消息丢失的处理分为预防丢失和丢失后兜底两层思路。针对两种不同的消息类型，处理策略有所不同：

**一、下单消息（核心业务消息）**

预防丢失：

1. 生产者配置 acks=all + enable.idempotence=true + retries=5，保证消息可靠写入 Kafka

2. 消费端使用手动 ACK，只有业务处理成功后才提交 offset，失败则不提交，Kafka 会重新投递

丢失兜底：

1. 如果消息发送失败，SeckillVoucherProducer 会立即回滚 Redis 预扣减的库存，保证 Redis 和 DB 状态一致

2. 如果消费失败，SeckillVoucherConsumer.afterConsumeFailure 也会回滚 Redis 库存，并记录对账日志

3. 如果消息延迟超过 10 秒才被消费，直接丢弃并回滚

最终兜底：

1. 定时对账任务（ReconciliationTaskServiceImpl）会扫描 Redis 扣减流水和 DB 对账日志的差异，发现不一致时删除 Redis 库存，保证最终一致性

**二、缓存失效消息（非核心通知消息）**

预防丢失：

1. 生产者有重试机制（最多 3 次，指数退避）+ 死信队列

2. 消费者有 DLQ 消费者专门处理发送失败的消息

丢失兜底：

1. 发布节点在发消息前已经同步删除了本地缓存和 Redis 缓存，所以发布节点不受影响

2. 其他节点如果收不到 Kafka 消息，本地缓存会一直持有脏数据，只有过期后才没有，在本项目中，只有秒杀优惠券这个不轻易修改的信息存进本地缓存中，创建后基本不会改。

3. 如果需要解决**本地缓存会一直持有脏数据**这个问题的话，有两个方法：缩短 TTL牺牲命中率换取一致性；改用版本号机制，给每个缓存数据加版本号，本地缓存存版本号，查询时先检查版本号是否和 Redis 一致，不一致则刷新。

**面试官：多个节点同时收到失效消息，会不会出现"惊群效应"？**

回答：不会出现惊群效应

首先发布节点自己会同步删除本地缓存和 Redis 缓存，然后才发 Kafka 消息

```
public void publishInvalidate(Long voucherId, String reason) {
    // 1. 先删除当前节点的本地缓存（同步，立即生效）
    seckillVoucherLocalCache.invalidate(seckillVoucherRedisKey.getRelKey());
    // 2. 再删除 Redis 缓存（同步，立即生效）
    redisCache.del(seckillVoucherRedisKey);
    redisCache.del(seckillVoucherStockRedisKey);
    redisCache.del(seckillVoucherNullRedisKey);
    // 3. 最后发 Kafka 消息通知其他节点
    invalidationProducer.sendPayload(topic, payload);

```

2.消费端使用分布式写锁：这把锁和数据更新时用的是同一把锁（UPDATE_SECKILL_VOUCHER_LOCK），保证不会同时有多个节点在更新同一张优惠券的数据、缓存失效和数据更新不会并发执行

3.删除操作是幂等的：无论是Caffeine的cache.invalidate(key)还是Redis 的 del都是幂等操作的，删一次和删十次效果一样，不存在并发安全问题

惊群效应真正危险的场景是：大量请求同时发现缓存失效，同时查数据库。但我们这里是"删除缓存"，不是"重建缓存"，删除后如果有人查询才会触发重建，而且重建时有分布式锁保护，不会出现惊群。

**面试官追问 1：为什么不在消费者端也实现重试？只在生产者端重试够吗？**

回答：项目中是有消费者重试机制的

1. Kafka 自身的重投机制：消费者使用手动 ACK，业务处理失败时抛出异常，不提交 offset。Kafka 检测到 offset 未提交，会自动把这条消息重新投递给消费者。这是 Kafka 协议层面保证的，不需要代码里写重试逻辑。

2. 消费失败后的补偿：消费者继承自 AbstractConsumerHandler，doConsume 抛异常时会触发 afterConsumeFailure，里面会做两件事回滚 Redis 预扣减的库存跟记录对账日志。

3. 然后异常继续向上抛，Kafka 重新投递，消费者再次处理。如果反复失败，消息会留在 Kafka 中等待人工介入。

**面试官追问 2：如果一个节点重启了，它能消费到之前丢失的消息吗？**

回答：分两种消息来看

**秒杀下单消息：**
1.下单消息的消费者使用的是默认的 groupId（配置里没有指定 groupId，Spring Kafka 会用 spring.kafka.consumer.group-id 的默认值）。节点重启后，groupId 不变，offset 也不变。

2.如果节点重启前没有提交 offset（因为消费失败抛了异常），Kafka会认为这条消息未被消费，重启后会重新投递。所以下单消息能消费到之前丢失的消息。

3.但如果节点重启前已经提交了 offset，那重启后不会重投，这条消息就丢了。不过这种情况下，doConsume 已经执行成功了，业务数据已经落库，消息丢失不影响数据一致性。
**缓存失效消息：**
1.缓存失效消息的 groupId 带 ${random.uuid}，节点重启后 groupId 变了，相当于新消费者，从新传入的消息开始消费，不会消费到之前的消息。

2.但正如我之前说的，发布节点在发消息前已经同步删除了缓存，其他节点会在TTL 过期后从 Redis 重建，不影响数据一致性。
**总结：**下单消息因为 groupId 固定，重启后能消费到未提交 offset 的消息；缓存失效消息因为 groupId 带UUID，重启后不能消费到历史消息，但有 TTL 兜底。"

**面试官追问 3：为什么不把本地缓存失效和 Redis 缓存失效分开处理？**

 回答：如果分开处理，需要维护两套消息，增加了复杂度。而且两者是同一个业务操作（"这张优惠券的数据变了"），语义上应该作为一个整体。

**面试官：Lua 脚本在 Redis 中是如何保证原子性的？Redis 执行 Lua 脚本时会阻塞其他命令吗？**

回答：

原子性保证：Redis 是单线程执行命令的。当客户端发送一个 Lua 脚本时，Redis会原子性地执行整个脚本，在执行过程中不会插入其他客户端的命令。也就是说，Lua 脚本就像一条命令一样，要么全部执行完，要么全部不执行。

```
-- 这三个操作在一个 Lua 脚本中，原子执行
redis.call('decrby', stockKey, 1)        -- 扣减库存
redis.call('sadd', seckillUserKey, userId) -- 记录用户已购买
redis.call('hset', traceLogKey, traceId, logEntry) -- 记录扣减流水
```

如果任何一个步骤失败（比如库存不足、用户已购买），脚本会提前返回错误码，整个操作回滚。

关于阻塞：会阻塞。Redis 执行 Lua 脚本期间，其他命令都要等待。但我们的脚本逻辑很简单（几个GET/INCRBY/SADD/HSET），执行时间在微秒级，对 Redis 的影响可以忽略不计。相比之下，如果不用 Lua 脚本，而是用多个单独的 Redis 命令，虽然不会阻塞，但不保证原子性——中间可能被其他命令插入，导致超卖。

**面试官：你提到"读写链路隔离"，具体是怎么实现的？读和写分别走的是什么路径？**

回答：

读链路：本地缓存 → Redis 缓存 → 布隆过滤器 → 数据库

写链路（扣库存）：Lua 脚本直接操作 Redis → 发送 Kafka 消息 → 消费者异步写数据库

隔离体现在：

1. 读不经过数据库：查询秒杀券详情只走缓存链路，不到数据库
2. 写不经过数据库：扣库存直接在 Redis 中完成，不碰 MySQL
3. 异步落库：扣库存成功后，通过 Kafka 异步通知消费者创建订单，消费者才写数据库

**面试官：如果 Lua 脚本执行过程中 Redis 节点宕机了，数据会丢失吗？你如何保证数据不丢失？**

回答：

**数据会丢失吗？**如果 Redis 主节点宕机且数据还没同步到从节点，确实会丢失部分库存扣减记录。这是我们项目的一个已知风险点。但项目通过对账补偿来保证最终一致性。

展开来讲

情况 1：单机 Redis

如果 Redis 宕机，Lua 脚本执行中断，库存扣减可能已经完成但用户购买记录和扣减流水可能没写入。这会导致数据不一致。

情况 2：Redis 集群/主从

Lua 脚本在主节点执行成功后，库存扣减是异步复制到从节点的。如果主节点宕机且数据还没同步到从节点，从节点提升为主后，丢失了刚扣减的库存。

**如何保证数据不丢失**

配置方法：

1. 开启 **AOF + appendfsync always** 可做到脚本执行后数据立即落盘，不丢数据

2. 更常用的方案是 **AOF everysec + 主从复制 + min-replicas-to-write**，可确保至少一个副本成功接收脚本修改，再返回客户端。

3. Redis Cluster 通过多副本复制，也能降低 Lua 执行过程中宕机导致的数据丢失风险。

项目的兜底：项目通过定时对账补偿来保证最终一致性

1. 每次执行库存扣减的Lua 脚本都会记录扣减流水（日志）包含：扣减前后的库存量、订单id、用户id、优惠券id等等

2. 会定期扫描 Redis 中的扣减流水，对比数据库中的订单

3. 如果发现 Redis 有扣减流水但 DB 没有对应订单（说明 Redis 宕机导致消息丢失），对账任务会删除 Redis 中对应的库存，保证最终一致性。

**面试官追问 1：为什么不直接用 Redis 的 MULTI/EXEC 事务？**

回答：

Redis 的 MULTI/EXEC 事务不支持条件判断和分支逻辑。我们的秒杀脚本需要根据时间、状态、库存、用户购买记录等多个条件做判断，只有满足所有条件才能扣减。而Lua 脚本可以在 Redis 单线程中执行复杂的逻辑（if/else、循环），并且保证原子性，这是事务做不到的。

**面试官追问 2：Lua 脚本执行时间过长会影响其他请求吗？**

回答：

会。Redis 是单线程的，Lua 脚本执行期间其他命令都被阻塞。但我们的脚本逻辑非常简单（几个 GET/INCRBY/SADD/HSET），执行时间在微秒级，对整体吞吐量的影响可以忽略。如果脚本逻辑复杂（比如大量循环），就应该拆分到数据库或用其他方式处理。
