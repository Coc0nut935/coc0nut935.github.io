 第一轮：基础概念（考察你对整体链路的理解）

  Q1. 秒杀请求从用户点击下单到最终订单落库，完整经历了哪些环节？把主链路串一遍。

  Q2. 为什么秒杀场景需要把库存预扣减放在 Redis 里，而不是直接操作数据库？数据库扛不住的根因是什么？

  Q3. 你的简历里提到"异步削峰解耦"，Kafka 在这条链路中具体解耦的是什么？为什么不直接同步写库？

**面试官：秒杀请求从用户点击下单到最终订单落库，完整经历了哪些环节？把主链路串一遍。**

回答：整条链路可以分为三个阶段

第一阶段：同步链路（用户感知到的）

1. 用户请求到 VoucherOrderController，先经过限流（令牌桶 + 滑动窗口，Lua 实现）

2. 进入 doSeckillVoucherV2，查询优惠券信息和库存，校验用户等级资格

3. 用雪花算法生成 orderId 和 traceId

4. 执行 Redis Lua 脚本 seckillVoucher.lua，在脚本内原子地完成：校验时间窗口 → 校验库存 → 校验一人一单（SISMEMBER）→扣减库存（INCRBY -1）→ 记录用户到已购集合（SADD）→ 写入扣减流水日志（HSET）

5. Lua 脚本返回成功后，构造 SeckillVoucherMessage，通过 Kafka Producer 异步发送消息

6. 立即返回 orderId 给用户

第二阶段：异步消费（Kafka Consumer）

7. SeckillVoucherConsumer 拉取消息，beforeConsume 先判断消息是否超时（>10s 丢弃并回滚 Redis）

8. doConsume 调用 createVoucherOrderV2，在事务中完成：一人一单二次校验 → 数据库扣减库存 → 插入 tb_voucher_order →插入 tb_voucher_order_router → 记录对账日志

9. 手动 acknowledge 提交 offset

第三阶段：异常处理

10. 如果消费失败，afterConsumeFailure 回滚 Redis 库存并记录对账日志；如果 Kafka 发送失败，afterSendFailure 同样回滚Redis

**面试官追问 1："返回给用户的 orderId 是怎么生成的？为什么不用数据库自增？"**

回答：雪花算法，在同步链路阶段就生成了，因为订单是异步落库的，如果用数据库自增，同步阶段就没有 orderId 返回给用户。雪花算法能保证全局唯一 + 趋势递增，而且不依赖数据库。

**面试官追问 2："你提到限流在最前面，如果限流拒绝了，还会执行 Lua 脚本吗？"**

回答：不会，限流是前置拦截，被限流的请求直接返回，不会进入秒杀链路。

**面试官：为什么秒杀场景需要把库存预扣减放在 Redis 里，而不是直接操作数据库？**

回答：

1. 性能差距：Redis 单线程内存操作 QPS 可以到 10 万+，MySQL 单表写入撑死几千 QPS。秒杀场景瞬时流量可能几万 QPS，直接打 DB 会把数据库连接池打满，拖垮整个系统。
2. 原子性：库存扣减需要「判断库存 > 0 → 判断未购买过 → 扣减库存 → 记录已购」这几步是原子的。用 Lua 脚本在 Redis 里一次性完成，不存在中间状态。如果在 MySQL 里做，要用分布式锁或 SELECT FOR UPDATE，并发越高锁竞争越激烈。
3. 读写隔离：Redis 承担秒杀的写链路，数据库承担最终的订单落库。通过 Kafka 解耦后，数据库面对的是平稳的异步消费流量，而不是瞬时尖峰。这个架构把热点写请求和冷数据持久化分开了。

**面试官追问 1："Redis 和 MySQL 的库存怎么保持一致？万一 Redis 扣了但 MySQL 没扣怎么**办？"

回答：这就是我简历里写的对账补偿机制。Lua 脚本在扣减时会往 traceLogKey 里写入一条扣减流水，记录 beforeQty、changeQty、afterQty、traceId。消费成功也会记录对账日志。定时对账任务会比对 Redis 库存和数据库订单数，发现不一致就补偿。

**面试官追问 2："Redis 宕机了库存数据不就丢了？"**

回答：是的，Redis 是内存存储，有丢数据的可能。但我的方案容忍这种风险，因为 Lua 脚本每笔扣减都写了流水日志（HSET traceLogKey），Redis 恢复后可以通过流水重建库存状态。而且真正的库存数据在 MySQL，Redis 只是缓存层。

**面试官追问 3："你提到 Lua 脚本保证原子性，那 Redis 单线程本身就串行执行命令，为什么还需要 Lua？"**

回答：单线程保证的是命令串行，但不保证多个命令之间的业务原子性。比如先 GET 判断库存，再 INCRBY 扣减，两条命令之间可能被其他客户端的命令插入。Lua 脚本是作为一个整体执行的，脚本内不会有其他命令插入，这才是真正的原子性。

**面试官：Kafka 在这条链路中具体解耦的是什么？为什么不直接同步写库？**

回答：Kafka 解耦的是秒杀请求处理和订单持久化这两个环节。

如果同步写库，每个秒杀请求都要等 MySQL 完成「扣库存 + 插订单 + 插路由表」这整个事务返回，MySQL 的 TPS 可能只有几千。用户请求来了要排队等锁、等事务提交，响应时间会从毫秒级退化到秒级，前端超时、连接堆积。

用了 Kafka 之后：

1. 同步链路只做 Redis 操作（微秒级），立即返回 orderId 给用户

2. 异步消费按 Kafka 分区有序消费，流量被削峰成平稳的消费速率，数据库不会被瞬时打爆

3. 即使消费暂时变慢，消息也只是积压在 Kafka 里，不会阻塞用户的秒杀请求

**面试官追问 1："消费端做了延迟超过 10 秒就丢弃，这个 10 秒是怎么来的？会不会误杀正常消息？"**

回答：这个阈值是根据业务场景设定的。秒杀请求有时间窗口，如果消息延迟了 10 秒还没被消费，说明消费积压严重。继续消费这些老消息价值不大，而且可能导致超卖（用户已经放弃但库存还在扣）。丢弃时会回滚 Redis 库存并记录对账日志，确保数据不丢。正常情况下消费延迟在毫秒级，10 秒足够覆盖。

第二轮：Redis Lua 原子扣减（考察实现细节）

  Q4. Lua 脚本扣库存时，除了扣减库存数，还做了哪些判断和操作？如果库存不足怎么处理？

  Q5. Redis 是单线程执行命令的，既然已经是单线程了，为什么还需要用 Lua 脚本来保证原子性？分开写多个命令不行吗？

  Q6. 如果 Lua 脚本执行到一半，Redis 宕机了，会出现什么问题？你的系统怎么应对这种情况？

**面试官：Lua 脚本扣库存时，除了扣减库存数，还做了哪些判断和操作？如果库存不足怎么处理？**

前置校验（4 层门禁）：

1. 时间窗口校验：用 redis.call('TIME') 取 Redis 服务端时间，和传入的 beginTime、endTime 比较，秒杀未开始返回10002，已结束返回 10003

2. 优惠券状态校验：status == 2（下架）返回 10011，status == 3（停售）返回 10012

3. 库存校验：GET stockKey，key 不存在返回 10004，stock <= 0 返回 10005

4. 一人一单校验：SISMEMBER seckillUserKey userId，已存在返回 10006 

业务操作（3 步写入）：

1. INCRBY stockKey -1 扣减库存

2. SADD seckillUserKey userId 将用户加入已购集合

3. HSET traceLogKey traceId logEntry 写入扣减流水日志，记录 beforeQty、changeQty、afterQty、orderId、traceId 等信息最后返回一个 JSON，包含 code=0（成功）和扣减前后的库存数量。库存不足时：脚本直接返回 code=10005，Java 侧判断 code != SUCCESS 后抛出 HmdpFrameException，不发送 Kafka 消息，用户直接收到"库存不足"提示。

**面试官追问 1："你用 redis.call('TIME') 而不是传入 Java 的时间，为什么？"**

回答：因为 Redis 是单线程执行 Lua 脚本的，用 Redis 服务端时间可以避免不同客户端机器时钟不一致的问题。如果用 Java 传入的本地时间，用户机器时间快 1 分钟，可能在秒杀还没开始时就通过了时间校验。

**面试官追问 2："你写了流水日志 HSET traceLogKey，这个流水具体有什么用？"**

回答：三个用途：第一，对账时比对每笔扣减是否都有对应订单；第二，回滚时作为依据，知道要恢复多少库存；第三，排查问题时可以按 traceId 追踪整条链路。这个 Hash 的 key 是 traceLogKey:{voucherId}，field 是 traceId，value 是包含完整扣减信息的 JSON。

**面试官追问 3："扣减流水的 TTL 是怎么设置的？为什么？"**

回答：设置为 优惠券结束时间 + 1 天。确保在优惠券有效期内流水不会过期，多留 1 天是为了给对账和补偿留出时间窗口。代码里是 Duration.between(now, endTime).getSeconds() + 86400。

**面试官：Redis 是单线程执行命令的，既然已经是单线程了，为什么还需要用 Lua 脚本来保证原子性？分开写多个命令不行吗？**

回答：

单线程 ≠ 原子性。Redis 单线程保证的是：任意时刻只有一个命令在执行，命令之间不会并行。但它不保证一组命令之间不被其他客户端的命令插入。

**面试官追问 1："那 Redis 事务 MULTI/EXEC 也能保证原子性，为什么不用事务？"**

回答：MULTI/EXEC 期间虽然不会有命令插入，但 WATCH 乐观锁在高并发下会频繁重试，性能不如 Lua。

**面试官追问 2："Lua 脚本执行时间太长会怎样？会阻塞其他请求吗？"**

回答：会的。Redis 执行 Lua 期间是阻塞的，其他所有命令都在排队。所以 Lua 脚本要尽量简短。我的脚本只做判断和简单读写，没有循环遍历，执行时间通常在微秒级，不会有阻塞问题。但如果你在 Lua 里做KEYS * 这种全量扫描，就会出问题。

**面试官追问 3："你的脚本里用了 redis.call 和 redis.pcall 的区别是什么？"**

回答：redis.call 执行出错会直接中断脚本并返回错误；redis.pcall 会捕获错误继续执行。

**面试官：如果 Lua 脚本执行到一半，Redis 宕机了，会出现什么问题？你的系统怎么应对这种情况？**

回答：这个问题要分两种情况讨论：
情况一：脚本执行前或执行中宕机Redis 的 Lua 脚本要么全部执行，要么全部不执行。因为 Redis 用单线程串行执行命令，脚本执行期间不会持久化（不会在脚本中间插入 RDB 快照或 AOF 写入）。如果执行中宕机，脚本的所有写操作都不会生效，等于这次扣减没发生。重启后库存数据不变，用户可以重新秒杀。这种情况是安全的。
情况二：脚本执行完，但还没返回结果给客户端时宕机脚本的写操作已经生效（库存已扣、用户已加入集合），但客户端没有收到成功响应。Java 侧会收到连接异常，不会发送 Kafka 消息。这时候库存被扣了但没有对应的订单。

我的应对机制：

1. Lua 脚本在扣减的同时写入了扣减流水 HSET traceLogKey，所以即使没有 Kafka 消息，这笔扣减也有记录

2. 定时对账任务会比对 Redis 扣减流水和数据库订单记录，发现「有扣减但无订单」的情况，自动回滚 Redis 库存

3. 回滚操作用另一个 Lua 脚本 seckillVoucherRollBack.lua，也是原子执行，并且有重试机制（最多 3 次，指数退避 + 抖动），回滚最终失败会写入 tb_rollback_failure_log 并触发告警

**面试官追问 1："对账任务多久跑一次？在这个时间窗口内，用户如果重复请求秒杀，能被拦住吗？"**

回答：对账任务我设的是定时调度，具体间隔取决于业务量级。用户重复请求在 Lua 脚本层就会被拦截，因为 SISMEMBER 会判断用户是否已在已购集合中。所以即使订单还没落库，用户再点秒杀也会返回 10006（已购买）。这个一人一单的控制不依赖数据库，是 Redis 层面就挡住的。

  第三轮：Kafka 异步消费与可靠性（考察消息不丢失、不重复）

  Q7. Kafka 消息从生产到消费，你做了哪些措施保证消息不丢失？acks=all 是不是就够了，还有哪些环节需要关注？

  Q8. 消费者拉取到消息后，如果业务处理成功但 ack 失败，会发生什么？反过来，ack 成功但业务处理失败呢？你怎么处理？

  Q9. 你简历提到"幂等键设计"，这个幂等键是什么，存在哪里，怎么判断是否重复消费？

  Q10. 如果消费速度跟不上生产速度，消息积压了，你会怎么处理？消费者延迟超过 10 秒就丢弃这个策略，在什么场景下是合理的？

**面试官：Kafka 消息从生产到消费，你做了哪些措施保证消息不丢失？acks=all 是不是就够了？**

回答：

acks=all 只是中一个环节，消息不丢失需要三个环节都保障：
生产端（我做了什么）：

1. acks=all：要求所有 ISR 副本确认写入成功才算成功，避免 leader 写完但 follower 还没同步时 leader 挂掉导致丢消息

2. retries=5：发送失败自动重试 5 次

3. enable.idempotence=true：开启幂等生产者，Kafka 会给每条消息分配序列号，broker 端按序去重，避免重试导致重复消

4. max.in.flight.requests.per.connection=5：配合幂等，允许最多 5 个未确认请求，兼顾吞吐和有序

5. 发送失败回调：afterSendFailure 里主动回滚 Redis 库存并记录对账日志，不丢数据

Broker 端：

1. ISR 副本机制，acks=all 保证消息写入所有同步副本

2. min.insync.replicas 通常设为 2，保证至少有 2 个副本同步

消费端（我做了什么）：

1. enable-auto-commit=false：关闭自动提交 offset

2. 手动 acknowledge：只有业务处理成功后才提交 offset。如果消费失败抛异常，offset 不会提交，下次拉取还能拿到这条消息

所以 acks=all 不够，它只解决了生产到 broker 的可靠性。消费端的处理失败、网络中断等场景还需要手动 ack + 业务侧幂等来兜底。

**面试官追问 1："你提到幂等生产者，它能完全解决重复问题吗？"**

回答：不能。幂等生产者只能保证单个 producer 实例在单个 session 内的幂等性。如果 producer 重启、或者跨 session 发送，Kafka 分配新的 producer ID，幂等就失效了。所以我还在消费端做了 @RepeatExecuteLimit 注解做二次幂等兜底。

**面试官：消费者拉取到消息后，如果业务处理成功但 ack 失败，会发生什么？反过来，ack 成功但业务处理失败呢？**

回答：

情况一：业务成功，ack 失败

消息的 offset 没有被提交。Kafka 认为这条消息还没被消费，消费者下次重启或 rebalance 后会重新拉取这条消息，导致重复消费。所以我的消费方法 createVoucherOrderV2 上有 @RepeatExecuteLimit 注解，用分布式锁 + Redis 标记做幂等，重复消费时直接抛异常，不会重复创建订单。

情况二：ack 成功，业务处理失败

这种情况不应该发生。看我的代码，onMessage 方法里是先调用 consumeRaw（执行业务逻辑），只有业务成功返回后才执行 acknowledgment.acknowledge()。如果 consumeRaw 抛异常，acknowledge() 不会执行，offset 不会提交。下次拉取会重新消费。

**面试官追问："你提到 @RepeatExecuteLimit 做幂等，它的实现原理是什么？"**

回答：它是一个自定义注解 + AOP 切面。切面里做了三层保护：

1. 先查 Redis 标记（repeatFlagName），如果已经处理过直接拒绝

2. 获取本地 ReentrantLock，防止同 JVM 内并发重复

3. 获取 Redisson 分布式公平锁，防止多实例并发重复

4. 业务执行成功后，在 Redis 里设置标记（SUCCESS_FLAG），标记的 key 包含消息的 UUID

**面试官追问："幂等键用的是什么？为什么用 UUID 而不是 orderId？"**

回答：幂等要防的是同一条消息的重复投递，UUID 是消息级别的唯一标识。而 orderId 是业务层的，理论上同一个订单可能触发多条不同消息（比如下单 + 取消），用 orderId 做幂等键可能误拦截。

**面试官：你简历里提到"幂等键设计"，这个幂等键是什么，存在哪里，怎么判断是否重复消费？**

回答：

**面试官：你们项目的幂等是怎么做的？**

回答：我们项目用的是自定义注解 + AOP 切面实现的分布式幂等框架。业务方法上加一个 @RepeatExecuteLimit 注解，声明业务名和幂等键（用 SpEL 表达式从方法参数中提取），切面会自动完成幂等检查和并发控制，业务代码完全不用关心幂等逻辑。

**面试官 追问：具体怎么做的**

回答：核心是三层防护：

1. 第一层，**Redis 标记快查**。每次执行前先查 Redis 里有没有 repeat_flag:{key} =success，有的话说明已经处理过，直接返回。这是最快的路径，大部分重复请求在这一步就被挡掉了。
2. 第二层，**本地锁**。用 Caffeine 缓存维护一个 ReentrantLock(fair=true)，同一 JVM内同一个幂等键只有一个线程能拿到锁，其他线程 tryLock 快速失败，避免打到 Redis 争抢分布式锁。
3. 第三层，**分布式锁**。拿到本地锁后，再通过 Redisson Fair Lock 获取分布式锁，保证集群环境下同一时刻只有一个节点执行。拿到分布式锁之后还会再查一次 Redis 标记（双重检查），因为可能在等锁的过程中别的线程已经执行完了。
4. **业务方法执行成功后，往 Redis 写一个 repeat_flag:{key} = success的标记**。我们秒杀场景下这个标记不设过期时间，因为幂等键用的是 Kafka 消息的 UUID，天然唯一，不需要过期。

**面试官追问：你简历写的 Redis 扣减流水 + 定时对账补偿，跟幂等是什么关系**

1. 他们是互补的，都是为了保证库存扣减的最终一致性
2. 通过幂等判断扣减操作是否已经进行过了
3.  Redis 扣减流水解决的是"Redis 扣了库存但 DB 没落库"的问题 — 每次扣减都记录一条 trace log

  **Q12. 对账发现 Redis 库存和数据库订单数不一致，有几种不一致的场景？分别怎么补偿？**

回答：①Redis 扣了，DB 没落 —— 最常见，也最重要；②Redis 没扣，DB 却有订单 —— 最危险，理论上不应该出现

场景①原因产生原因有三种：

a) Kafka 消息丢失。 生产者发送成功但 Kafka broker 宕机，消息没落盘，消费者永远收不到。
b) 消费者处理失败。 消费者收到消息后 createVoucherOrderV2 抛异常（比如 DB 连接超时），事务回滚，DB 没有订单。虽然afterConsumeFailure 会触发 Redis 回滚，但如果回滚本身也失败了，就变成了 Redis 扣了、DB 没落、回滚也没成功。
c) 消息延迟超过 10 秒被丢弃。 消费端 beforeConsume 检测到消息延迟超过阈值，直接丢弃并回滚Redis。但回滚同样可能失败。

检测方式： Redis trace log 中存在 logType=DEDUCT 的流水，用 orderId + traceId 去tb_voucher_reconcile_log 查不到对应记录。但要排除正常延迟——只有超过 12 秒保护窗口（10 秒消费超时 + 2 秒缓冲）仍然查不到，才判定为真实异常。

补偿方法：删除 Redis 库存 key，不做 INCRBY +1。原因是不确定到底有几条流水处于这种状态，直接删 key 让下次请求从 DB 重新加载，以DB 库存为准。同时清理掉这条脏流水，避免后续对账重复命中。

场景②原因产生原因有两种：

a) 对账回填（backfill）误操作。 阶段二对账时从 DB 往 Redis 补全日志，如果补全逻辑有 bug，可能造成 Redis看起来"没有"流水但实际是被补全覆盖了。
b) 人工直接操作 DB。 运维或开发直接在数据库里插入了订单记录，绕过了 Redis 扣减流程。

检测方式：查询所有 DB 订单，检查 Redis trace log 中是否有对应的 DEDUCT 流水。没有就是异常。

补偿方法：

这个最危险，因为库存可能已经超卖。需要拿 DB 的订单数量反推实际库存：实际库存 = 初始库存 - DB已完成订单数，然后用这个值覆盖 Redis 库存 key。同时标记订单为 ABNORMAL，人工介入排查原因。
